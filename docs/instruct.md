# CustomQA 项目汇报说明

## 1. 项目定位

这个模块是在原有 `SearchEngine` 项目基础上扩展出来的自定义问答搜索部分。

原项目更偏向：

- 商品搜索
- 保险问答搜索

我这次新增和改造的是：

- 支持自定义问答数据
- 支持一个问题对应多个不同质量的答案
- 支持基于问题、答案、主题的综合检索
- 支持答案质量分参与排序
- 支持结果去重与详情页多答案展示

一句话概括：

**这是一个面向自定义专题问答库的 Elasticsearch 检索与排序实验项目。**

---

## 2. 业务目标

我希望这个模块解决的问题是：

1. 能把自己整理的问答数据写入 ES
2. 能按关键词检索相关问题和答案
3. 不只是按文本匹配排序，还要把答案质量分纳入排序过程
4. 同一个问题下有多个答案时，既能在结果页快速浏览，也能在详情页集中比较

所以整个设计重点不是“只搜问题”，而是：

**把“问题 + 单条答案”作为一个候选结果进行搜索与排序。**

## 3. 数据设计

### 3.1 源数据文件

当前使用的数据文件是：

- `search-engine/data/example.json`

数据结构为：

```json
{
  "topic": "Java零基础到入门",
  "questions": [
    {
      "id": "q1",
      "question": "== 和 equals() 到底有什么区别？",
      "answers": [
        {
          "id": "a1",
          "answer": "答案内容",
          "answer_quality": 9
        }
      ]
    }
  ]
}
```

这套结构的含义是：

- 一个专题 `topic`
- 下面有多个问题 `questions`
- 每个问题下再挂多个答案 `answers`
- 每个答案都带一个人工标注的 `answer_quality`

这样设计的好处是：

- 人工维护直观
- 同学协作时不容易把问答对应关系写乱
- 很适合表达“同一问题有多种质量不同的答案”

### 3.2 ES 搜索文档

虽然源文件是嵌套结构，但写入 Elasticsearch 时并不是整块存进去，而是展开成多条搜索文档。

每条 ES 文档对应：

- 一个问题
- 下面的一条具体答案

展开后的核心字段包括：

- `id`
- `qid`
- `topic`
- `question`
- `answer`
- `answerQuality`

这样做的原因是：

- 文本相关性来自 `question` 和 `answer`
- 排序信号来自 `answerQuality`
- 搜索单位直接就是“当前问题 + 当前答案”

### 3.3 技术对象与索引

当前自定义问答部分的核心 Java 实体是 `CustomQARecord`，字段包括：

- `id`
- `qid`
- `topic`
- `question`
- `answer`
- `answerQuality`

它对应的代码位置是：

- `src/main/java/com/example/demo/pojo/CustomQARecord.java`

当前自定义问答数据写入的 ES 索引名是：

- `custom_qa`

对应的索引常量位置是：

- `src/main/java/com/example/demo/constants/EsIndexNames.java`

这里我没有把问题和答案拆成两个索引，而是直接把“问题 + 单条答案”合成一个搜索文档。这样做的技术优势是：

- 避免跨索引关联查询
- ES 查询时可以直接基于单条文档打分
- `answerQuality` 可以天然作为当前候选答案的排序信号

### 3.4 ID 生成规则

为了避免重复导入时产生脏数据，当前不是直接拿原始 `q1`、`a1` 去做 ES 主键，而是会自动生成稳定 id。

生成逻辑是：

- `qid = topic + "-" + questionId`
- `id = qid + "-" + answerId`

如果原始字段缺失，就按问题下标、答案下标补默认值。

这样做有两个好处：

- 重复执行导入接口时，ES 文档会被稳定覆盖，而不是越写越多
- 结果页和详情页可以稳定通过 `qid` 关联同一问题下的所有答案

## 4. 核心实现思路

### 4.1 数据导入

导入流程是：

1. 读取 `data/example.json`
2. 解析顶层 `topic`
3. 遍历 `questions`
4. 再遍历每个问题下的 `answers`
5. 每个答案生成一条 `CustomQARecord`
6. 批量写入 `custom_qa` 索引

也就是说：

- 源 JSON 是“一个问题带多个答案”
- ES 里是“每个答案一条搜索记录”

从代码链路上看，导入流程是：

1. 控制器接口 `/writeCustomQA` 或 `/writeQA` 进入后端
2. 调用 `ContentService.writeQAContent()`
3. `ContentService` 再调用 `JsonParseUtil.parseCustomQaJson(...)`
4. 解析结果转成 `List<CustomQARecord>`
5. 通过 ES `BulkRequest` 批量写入 `custom_qa`

这样设计的原因是：

- 批量写入比逐条写入更适合 ES
- 结构转换集中在解析层，服务层只负责索引写入和查询
- 控制器保持很薄，便于后期扩展

### 4.2 自动加载机制

这个模块支持自动准备数据：

- 服务启动时会尝试检查 `custom_qa`
- 第一次搜索时也会再次检查
- 如果索引不存在或没有数据，就自动导入 `data/example.json`

这样做的好处是：

- 使用时不一定要手动先点导入接口
- 减少答辩演示时的步骤

技术上它有两层保护：

- `@PostConstruct` 启动检查
- `searchQA(...)`、`searchAnswer(...)` 查询前再次检查

所以即使启动阶段没准备成功，只要第一次真正搜索，系统也会再次尝试自动导入。

### 4.3 搜索逻辑

搜索接口不是只搜问题标题，而是综合考虑：

- `question`
- `answer`
- `topic`

并且权重不同：

- `question` 权重最高
- `answer` 次之
- `topic` 最低

原因很简单：

- 用户大多数时候首先是按问题找内容
- 但答案正文里也可能包含更丰富的命中信息
- 主题只是辅助信息

接口层面，当前搜索入口是：

- `POST /queryse`

请求参数包括：

- `keyword`
- `pageNo`
- `pageSize`
- `deduplicate`

其中：

- `keyword` 是检索词
- `pageNo` 和 `pageSize` 控制分页
- `deduplicate` 控制是否按问题去重

从 ES 查询实现看，当前使用的是 `multiMatchQuery`，主要字段和权重为：

- `question = 3.0`
- `answer = 2.0`
- `topic = 1.2`

### 4.4 排序逻辑

排序不是只用 ES 默认文本相关性，而是做了“文本相关性 + 质量分加权”。

可以概括成：

**最终分数 = 文本相关性 + answerQuality 加权**

这样做有两个好处：

- 搜索词和问题、答案越相关，结果越靠前
- 当多条结果文本相关性接近时，质量更高的答案会排得更前

技术实现上，当前排序并不是 Java 内存里手工先排文本再加分，而是：

1. 先构建文本查询
2. 再用 ES 的 `scriptScoreQuery`
3. 在 `_score` 基础上叠加 `answerQuality * 0.2`

所以当前打分公式可以更具体地写成：

**finalScore = _score + answerQuality * 0.2**

这里的 `_score` 是 ES 文本相关性分数，来自：

- `question`
- `answer`
- `topic`

这里的 `answerQuality * 0.2` 是人为加入的排序增强项。

这个系数没有设得太大，目的就是：

- 让文本相关性依然是主体
- 让质量分在“文本差不多”的情况下发挥区分作用

这也符合这个数据集的特点：

- 数据不是单一标准答案集
- 而是同一问题有多个质量不同的候选答案

### 4.5 布尔查询

当前搜索页支持简单布尔表达式：

- `AND`
- `OR`
- `NOT`

例如：

- `Java AND 接口`
- `线程 OR 并发`
- `集合 NOT List`

这使得它不只是普通关键词检索，而更像一个真正的搜索实验项目。

技术实现上，当前不是依赖 ES 原生查询语法字符串直接透传，而是先在后端做了一层解析：

- 包含 `AND` 时，拆分后构建 `must`
- 包含 `OR` 时，拆分后构建 `should`
- 包含 `NOT` 时，拆分后构建 `mustNot`

这样可以保证表达式逻辑更可控，也便于在每个子项里继续复用多字段加权检索。

### 4.6 问题去重实现

问题去重是当前实现里比较重要的一个技术点。

如果不开去重：

- ES 会直接返回所有命中的答案文档
- 同一问题可能出现多条结果

如果开启去重：

1. 后端先放大 ES 召回窗口
2. 按 `qid` 对结果分组
3. 每组只保留一个“代表答案”
4. 代表答案的选择规则是：
   - 先比较 `_searchScore`
   - 如果 `_searchScore` 相同，再比较 `answerQuality`
5. 最后再按页码分页

这样做的意义是：

- 开启去重时，结果页更像“问题列表”
- 关闭去重时，结果页更像“完整候选答案列表”
- 两种模式都能复用同一套索引和同一套排序逻辑

## 5. 结果页与详情页设计

### 5.1 结果页

结果页的搜索单位本质上是“答案文档”。

因此如果不做额外处理，会出现：

- 同一个问题下多个答案都命中
- 同一个问题在结果页出现多次

这是合理的，因为当前比较的是“不同答案候选”。

结果页每条卡片会展示：

- 编号
- 问题所属标识
- 主题
- 质量分
- 问题文本
- 当前命中答案摘要

### 5.2 详情页

点击结果后，进入 `/answer/{qid}` 详情页。

详情页不是只显示当前这一条答案，而是：

1. 根据 `qid` 找到同一个问题下的所有答案
2. 按 `answerQuality` 从高到低排序
3. 顶部显示最佳答案摘要
4. 下方展示完整候选答案列表

这样用户既能快速看最佳答案，也能比较不同答案之间的质量差异。

## 6. 创新点

我这部分的创新主要有 5 个。

### 6.1 自定义问答数据建模

没有继续沿用原保险数据集的双文件格式，而是改成了：

- `topic`
- `questions`
- `answers`
- `answer_quality`

这更适合自定义专题问答库。

### 6.2 多字段联合检索

搜索不是只匹配问题，而是联合匹配：

- 问题
- 答案
- 主题

这让召回结果更完整，也更适合专题问答场景。

### 6.3 质量分参与排序

答案的人工质量分不只是展示信息，而是真正参与了排序。

这比“纯文本相关性排序”更符合问答场景。

### 6.4 支持布尔查询

系统支持 `AND / OR / NOT`，可以更灵活地表达检索需求。

### 6.5 问题去重开关

这是后来新增的一个比较实用的功能。

搜索页增加了“合并同问题结果”开关：

- 开启时：按 `qid` 去重，每个问题只保留最优命中的答案
- 关闭时：保留所有命中答案，适合观察不同答案之间的排序差异

这个功能的意义在于：

- 普通用户更适合看去重后的问题列表
- 实验和调试时更适合看完整答案候选列表

也就是说：

**同一套系统同时兼顾了“用户视角”和“实验视角”。**

### 6.6 UI 设计风格优化

除了后端检索和排序逻辑，我还对问答模块的前端界面做了重设计。

这部分 UI 不是直接沿用原项目的默认样式，而是：

- 参考了开源项目里的编辑式信息布局
- 视觉风格上借鉴了 Claude 类产品常见的留白、圆角卡片和暖色中性色体系
- 再结合当前问答搜索场景做了本地化调整

所以它并不是“完全照搬”，而是：

- 在开源界面风格基础上进行仿照和再实现
- 目标是让搜索页和详情页更适合阅读长文本答案

这部分优化的意义在于：

- 搜索结果的信息层级更清楚
- 问题、答案摘要、质量分之间的视觉关系更明确
- 详情页更适合比较同一问题下多个候选答案

答辩时可以把这部分概括为：

**前端界面参考开源项目做了 Claude 风格的阅读型改版，用来提升问答检索场景下的信息可读性。**

## 7. 当前实现的页面功能

### 7.1 搜索页

搜索页目前支持：

- 输入关键词搜索
- 支持布尔表达式
- 查询示例一键点击
- 显示当前查询词
- 显示返回结果数量
- 结果去重开关

前端技术上：

- 页面模板是 `templates/se.html`
- 使用 Vue2 + axios
- 前端点击搜索时调用 `POST /queryse`
- 结果页点击条目后跳转 `/answer/{qid}`

### 7.2 详情页

详情页目前支持：

- 展示问题文本
- 展示主题
- 展示最佳答案质量分
- 展示同问题下全部答案
- 按质量分排序

详情页技术上：

- 页面模板是 `templates/answer.html`
- 详情控制器入口是 `GET /answer/{qid}`
- 后端会先查同一 `qid` 的所有答案
- 再按 `answerQuality` 降序排序
- 第一条答案作为“最佳答案摘要”展示在顶部

## 11. 老师可能追问的技术点

### 11.1 为什么不拆成“问题索引 + 答案索引”来做搜索

因为当前最核心的是排序，而排序要同时依赖：

- 问题相关性
- 答案相关性
- 当前答案质量分

如果拆成两个索引，搜索阶段要么只能先搜问题、后查答案，要么要做更复杂的关联处理。对这个实验项目来说，直接把“问题 + 单条答案”合成一个搜索文档更直接、更稳定。

### 11.2 为什么详情页还要按 `qid` 聚合

因为结果页解决的是“找到最相关的候选结果”，而详情页解决的是“查看同一问题下全部答案并比较质量差异”。

所以：

- 搜索页按候选答案排序
- 详情页按问题聚合展示

这两个页面承担的是不同职责。

### 11.3 为什么质量分不是主排序，而是辅助排序

因为搜索系统首先还是要保证“搜得对”。

如果质量分权重过大，会出现：

- 文本其实不太相关
- 但因为质量高被硬排到前面

所以当前把质量分做成辅助项，而不是主导项，这样更符合搜索业务逻辑。
